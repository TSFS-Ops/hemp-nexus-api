# Phase 1 — Unknown-Counterparty Facilitation Cases (foundation)

Scope explicitly excluded from Phase 1 (deferred to later batches): approved-email outreach, duplicate / do-not-contact / sanctions checks, compliance escalation logic, SLA cron + amber/red overdue notifications, management reporting dashboard, CSV export, audit-pack export PDF.

Phase 1 success condition (your wording): user submits a facilitation case → stored in `facilitation_cases` → admin sees it in queue → admin assigns owner → admin changes status → every material action is audited → requesting user sees a simple milestone status.

## 1. Database — new tables

### `public.facilitation_cases`

- `id uuid pk`
- `case_number text unique` (human-readable, e.g. `FAC-2026-000123`, generated by sequence + trigger)
- `requesting_org_id uuid not null → organizations(id)`
- `requesting_user_id uuid not null → auth.users(id)`
- `trade_request_id uuid not null → trade_requests(id)`  *(required link)*
- `poi_engagement_id uuid null → poi_engagements(id)`  *(optional, set if/when one exists)*
- Intake (per Section 2 of spec):
  - `counterparty_legal_name text not null`
  - `counterparty_trading_name text null`
  - `counterparty_country text not null` (ISO-2)
  - `counterparty_city text null`
  - `counterparty_website text null`
  - `counterparty_email text null`
  - `counterparty_phone text null`
  - `counterparty_contact_name text null`
  - `product_or_commodity text not null`
  - `role text not null check (role in ('buyer','seller'))`
  - `estimated_value_amount numeric(20,2) not null`
  - `estimated_value_currency text not null` (ISO-4217)
  - `urgency text not null check (urgency in ('low','normal','high','critical'))`
  - `reason text not null`
  - `how_user_knows_counterparty text not null` + `how_user_knows_notes text null`
  - `permission_to_contact boolean not null`
  - `user_declaration_accepted boolean not null`
  - `user_declaration_accepted_at timestamptz not null`
- Lifecycle:
  - `internal_status text not null default 'new'` (CHECK = the 17 internal statuses from Section 4)
  - `user_facing_status text not null` (derived on write by trigger from internal_status, per Section 6 mapping)
  - `case_owner_id uuid null → auth.users(id)`
  - `closing_reason text null`
  - `final_outcome text null` (CHECK = the 11 outcomes from Section 12)
  - `linked_organization_id uuid null → organizations(id)`
  - `created_at`, `updated_at`, `closed_at`
- Validation trigger (not CHECK) enforces:
  - status transitions match the lifecycle whitelist (Section 4)
  - cannot leave `new` without an `case_owner_id`
  - cannot enter terminal status without `final_outcome` + `closing_reason`
  - `user_facing_status` always derived, never client-writable

### `public.facilitation_case_evidence`

- `id`, `case_id → facilitation_cases(id) on delete cascade`
- `storage_path text not null` (private bucket `facilitation-evidence`)
- `original_filename`, `mime_type`, `size_bytes`, `uploaded_by`, `created_at`

### `public.facilitation_case_events` (append-only audit, scoped to this feature)

- `id`, `case_id`, `actor_user_id`, `action text not null`, `from_status text null`, `to_status text null`, `payload jsonb not null default '{}'`, `created_at`
- Canonical action prefix `facilitation_case.*` (e.g. `facilitation_case.created`, `.assigned`, `.status_changed`, `.note_added`, `.evidence_uploaded`, `.contact_attempted`, `.closed`). Locked by a new prebuild guard `scripts/check-facilitation-case-audit-names.mjs` (drift-protected, same pattern as `check-ai-review-audit-names.mjs`).

### Storage

- New private bucket `facilitation-evidence` via `supabase--storage_create_bucket`. RLS on `storage.objects`: requesting org members + platform_admin/compliance_admin read; uploader inserts; immutable thereafter.

### Grants & RLS (mandatory)

For each new table:

- `GRANT SELECT, INSERT, UPDATE ON ... TO authenticated`
- `GRANT ALL ON ... TO service_role`
- `ENABLE ROW LEVEL SECURITY`
- Policies:
  - **facilitation_cases**: requester's org members can `SELECT` their own cases; requester can `INSERT` for own org; `platform_admin` + `compliance_admin` + `case_owner_id = auth.uid()` can `SELECT/UPDATE` (UPDATE further scoped to allowed columns via edge function — direct UPDATE from client blocked for status/owner/outcome).
  - **facilitation_case_events**: read for org members (filtered set — internal-only events redacted by edge function, not by direct SELECT; Phase 1 keeps it simple by routing user reads through `get-facilitation-case` edge function); insert service_role only.
  - **facilitation_case_evidence**: same org-scoped read; insert by requester or case owner.

## 2. Edge functions (all `verify_jwt = false`, validate JWT in code, Zod input)


| Function                            | Purpose                                                                                                           |
| ----------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| `create-facilitation-case`          | Validates intake; inserts case; emits `facilitation_case.created` event                                           |
| `update-facilitation-case-status`   | Admin-only; enforces lifecycle transitions; writes `from_status`/`to_status` event                                |
| `assign-facilitation-case`          | Admin-only; sets `case_owner_id`; emits `.assigned` event                                                         |
| `add-facilitation-case-note`        | Owner/admin; appends note as event payload                                                                        |
| `upload-facilitation-case-evidence` | Returns signed upload URL for `facilitation-evidence` bucket; registers row                                       |
| `get-facilitation-case`             | Single-case fetch with role-aware redaction (requester gets milestone view; admin gets full event log)            |
| `list-facilitation-cases`           | Admin queue listing with filters: status, owner, urgency, country (no pagination beyond default limit in Phase 1) |


No outreach/send paths. No links to `send-transactional-email`. A `scripts/check-facilitation-no-send-path.mjs` prebuild guard greps these functions for `send-transactional-email|resend|twilio|sendgrid|smtp` and fails the build if matched.
