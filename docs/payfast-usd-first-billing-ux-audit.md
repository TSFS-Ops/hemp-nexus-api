# PayFast USD-First Billing UX and Payment Metadata Audit — Inspection First

Status: read-only inspection. No checkout, ITN, Paystack, pricing, ledger, or wallet-crediting logic was changed while producing this document.

## 0. Scope and constraints

Goal: identify what can safely change on the Izenzo side now so customer-facing billing/checkout is USD-first and clean, while PayFast still processes ZAR under the hood and PayFast's own USD/multi-currency capability is unconfirmed. Technical ZAR amount, FX rate, and settlement metadata may still be retained in the background for audit, reconciliation and PayFast processing.

Not changed in this pass: PayFast, Paystack, wallet crediting, PayFast ITN logic, Paystack webhook logic, ledger logic, pricing, metadata storage, external FX APIs, PayFast USD-checkout assumptions, historical payment rows.

## 1. Billing page customer UI

Two separate customer-facing billing surfaces exist and are in different states:

- `src/pages/Billing.tsx` (route `/billing`). Paystack-only checkout via the `token-purchase` edge function. No PayFast option is rendered here. Copy is already USD-clean: "Prices in USD ($10 per credit). PayFast charges the equivalent ZAR amount at the rate Izenzo locks in when checkout starts." (customer-facing, mentions PayFast/ZAR in passing but does not show a live rate or ZAR figure) and "Payments processed securely by Paystack. All prices shown and charged in USD." Admin-only additions on this page: `AdminPayfastPricingReview` (gated by `isAdmin`), `PayfastSandboxTestButton`, `PayfastLiveSmokeTestButton` (all documented in-code as "never a customer surface").
- - `src/components/desk/billing/BillingOverview.tsx` (rendered inside `src/pages/Desk.tsx` at `/desk/billing`). This is the second, actively-used customer surface. It embeds `PaymentMethodPicker`, which is the actual PayFast/Paystack selector.
 
  - Finding (customer-facing, needs fixing): `src/components/desk/billing/PaymentMethodPicker.tsx` shows ordinary (non-admin) customers the computed PayFast ZAR amount on the button itself ("Pay R873 via PayFast") and an explicit FX-rate disclosure line: `` `${usd} · PayFast amount: R{zar} · Rate used: $1 = R{rate}` `` (`data-testid="payment-method-fx-note-${packageId}"`). This is rendered whenever `showPayfast` is true, which depends only on PayFast availability and package eligibility — it is NOT gated by `isAdmin`. This directly contradicts the "USD-first and clean, hide ZAR/rate" goal and is the single biggest customer-UI item to fix.
 
  - Paystack visibility on this same picker is already correctly hidden: `showPaystack = PAYSTACK_PUBLIC_ENABLED || isAdmin`, and `PAYSTACK_PUBLIC_ENABLED = false` in `PaymentMethodPicker.tsx`. When shown only to an admin, the button is explicitly labelled "[Admin only] Pay {usd} via Paystack" with `data-admin-only="true"` and a title tooltip "Admin-only / internal — not visible to customers".
 
  - No sandbox/smoke/test wording, old pricing, or confusing legacy copy was found on either customer surface; the admin-only test buttons render to null for non-admins per their own doc comments and are excluded from this classification.
 
  - Classification: Billing.tsx = clean (A, no change needed for USD-first goal). BillingOverview.tsx / PaymentMethodPicker.tsx = customer-facing ZAR + FX-rate exposure that needs to be hidden or reworded (B — real feature, currently over-exposes settlement detail to customers).
 
  - ## 2. PayFast checkout initiation
 
  - `supabase/functions/_shared/payments/payfast-public-checkout.ts` (`buildPayfastPublicCheckout`, entry point `supabase/functions/payfast-checkout-public/index.ts`) is the customer checkout path. Confirmed:
 
  - - USD price is the source of truth: `pkg.price_usd` comes from `getPayfastCustomerPackage(input.packageId)`.
    - - The ZAR amount is computed before the PayFast redirect: `computeZarAmount(pkg.price_usd, rate)`, and checkout is rejected with `fx_rate_missing` if no admin-set rate exists (`admin_settings.payfast_usd_zar_rate`, read via the `usePayfastPublicAvailability` probe / edge function).
      - - ZAR is what is actually sent to PayFast (`amount: amountStr`, PayFast is ZAR-native; there is no USD PayFast integration today).
        - - A full metadata snapshot is written to BOTH `token_purchases.metadata` and a `credits.purchase_initiated` `audit_logs` row at initiation time, including: `price_usd`, `amount_usd`, `usd_zar_rate`, `fx_rate_locked_at`, `fx_rate_source`, `amount_zar`, `price_zar`, `provider`, `provider_reference` (=`m_payment_id`), `package_id`, `token_amount`/`credits`, `mode`, `currency`. This snapshot is locked at checkout start so later admin rate changes cannot retroactively change an in-flight purchase.
         
          - No FX API is imported here (confirmed by the file's own "NON-GOALS" comment and by `_shared/fx.ts` not being referenced) — consistent with the "do not revive external FX APIs" constraint.
         
          - ## 3. PayFast return and cancel pages
         
          - - `src/pages/desk/billing/PayfastReturn.tsx`: polls `token_purchases` by `provider_reference` only; never calls a credit RPC or `token-purchase/verify`, never writes to `token_purchases` (confirmed by both direct read and the existing test `src/tests/payfast-phase-2j-customer-rollout.test.ts`, which asserts exactly this). Copy is clean: "Your wallet is credited only after PayFast confirms the payment with Izenzo (verified ITN)." It shows credits added and new balance, plus the internal `provider_reference` token (not a ZAR amount, not a rate) as "Reference: <id>" — acceptable, not confusing.
            - - `src/pages/desk/billing/PayfastCancel.tsx`: makes no Supabase calls at all (test-enforced: `expect(CANCEL_PAGE_SRC).not.toMatch(/supabase/i)`), so it cannot credit and does not expose any settlement detail.
             
              - Both pages are already read-only, already USD/ZAR-neutral, and require no changes for the USD-first goal.
             
              - ## 4. Purchase history
             
              - Customer-facing: `src/components/desk/billing/PurchasesList.tsx`, powered by `supabase/functions/list-org-purchases/index.ts`, scoped to the caller's own org. It shows: credits, `provider` badge (PayFast/Paystack), a single amount line — ZAR wording only for PayFast rows ("· ZAR via PayFast", no figure or rate shown) vs "· $X.XX USD via Paystack" for Paystack rows — status badge, created date, and reference (provider_reference for PayFast, paystack_reference for Paystack). This is reasonably clean already; the "ZAR via PayFast" tag is a mild PayFast-currency disclosure but carries no rate or ZAR figure.
             
              - Admin-facing (the audit item closed in this pass): there is no dedicated admin UI that lists individual `token_purchases` rows directly. Two admin surfaces exist instead:
             
              - - `src/components/admin/AdminTokenManagement.tsx` — shows only aggregate `token_balances` per organisation (current balance, minimum required) plus a manual top-up dialog. No per-purchase rows, no provider/reference/FX data at all.
                - - `src/components/admin/AdminRevenuePanel.tsx` (HQ → Revenue & sales) — this is the actual admin transaction/metadata view. It is built from `audit_logs` rows where `action = 'credits.purchased'` (the canonical settlement event), with `token_ledger` rows (`action_type = 'credit_purchase'`) as a manual-reconciliation safety net, plus `credits.purchase_initiated` rows for a "Pending settlement" panel. Per-org timeline columns rendered: When, Organisation, Source, Package, Credits, USD, Currency, Legacy ZAR, Legacy FX, Reference. CSV export carries the same fields plus `audit_log_id`/`ledger_id`.
                 
                  - Field-by-field classification against the requested list, based on what `payfast-public-checkout.ts` (initiation) and `payfast.ts` (`processPayfastItn`, settlement) actually write, and what `AdminRevenuePanel.tsx`'s parser (`purchaseFromAuditLog`) actually reads:
                 
                  - - USD package price / `amount_usd` / `price_usd`: stored at initiation (both `token_purchases.metadata` and the initiation `audit_logs` row). NOT stored on the settlement `credits.purchased` audit row written by `payfast.ts` (that row only carries `price_zar` and `amount_gross_zar`). Because the panel's `isNativeUsd` check requires `meta.price_usd > 0`, and the settlement row never has it, every completed PayFast purchase is classified as legacy-ZAR with `amount_usd` forced to `0`. Result: stored but not visible in admin UI at settlement time; PayFast revenue is invisible in the panel's headline Revenue total, the time-series chart, and the Top Buyers ranking (all keyed off `amount_usd`).
                    - - ZAR amount charged / `amount_zar` / `price_zar`: stored at initiation and at settlement (`price_zar`, `amount_gross_zar` on the `credits.purchased` row). Visible in the admin UI as the "Legacy ZAR" column.
                      - - USD/ZAR rate used / `fx_rate_locked_at`: stored at initiation as `usd_zar_rate` and `fx_rate_locked_at` (both in `token_purchases.metadata` and the initiation audit row). NOT carried onto the settlement `credits.purchased` row at all — `payfast.ts`'s success-audit metadata block has no rate field. Separately, even where a rate value did exist, `AdminRevenuePanel.tsx` reads `meta.fx_rate` / `meta.legacy_fx_rate`, but PayFast writes `usd_zar_rate` — a field-name mismatch. Net effect: the "Legacy FX" column is always blank ("-") for PayFast rows. Classification: stored (at initiation only) but not visible in admin UI.
                        - - Provider: stored and visible (`provider: "payfast"` on every row; badge/column rendering is correct).
                          - - Provider reference / PayFast reference (`provider_reference`, `pf_payment_id`): stored throughout (initiation metadata, `token_purchases.provider_reference` column, and every ITN-path audit row). NOT visible in `AdminRevenuePanel.tsx`: the panel's `payment_reference` field reads `meta.payment_reference` or `meta.reference`, but PayFast never writes either key (it writes `provider_reference` and `pf_payment_id` instead). Result: the "Reference" column and the CSV `payment_reference` column are blank for every PayFast row, even though the reference data exists in the same metadata object the panel already queries (`raw_metadata` is kept in memory on `PurchaseEnriched.raw_metadata` but there is no expand/detail view that renders it). Classification: stored but not visible in admin UI.
                            - - Payment status: implicit only — the panel only ever queries the settled (`credits.purchased`) event, so every row it shows is by definition completed. Pending/failed/cancelled PayFast purchases are not shown here at all except indirectly via the "Pending settlement" panel (which only catches initiated-but-never-settled rows) and via `admin_risk_items` rows written for rejected ITNs (existence of a general `admin_risk_items` viewer elsewhere in admin was not confirmed in this pass). Classification: partially visible — no admin timeline of pending/failed/cancelled PayFast token_purchases rows exists today.
                              - - `package_id`: stored and correctly visible (initiation, settlement, and the "Package" column).
                                - - `token_amount`/credits: stored and correctly visible (`credits_added` on settlement, rendered as "Credits").
                                  - - Mode (`live`/`sandbox`): stored in both initiation and settlement metadata (`mode` field). NOT rendered as a column anywhere in `AdminRevenuePanel.tsx` — present in `raw_metadata` but not surfaced in the UI.
                                    - - Audit log linkage: visible at the row level (`source: "audit_log"` badge, `audit_log_id` in CSV export) but there is no click-through to view the underlying `audit_logs` row in place; an admin would need to separately open `AdminAuditLogs.tsx` and search by id.
                                      - - Wallet/ledger linkage: partially visible — ledger-sourced rows show `source: "ledger"`/`"manual"` and carry a `request_id`, but this is not rendered as a visible column; audit_log-sourced rows (the majority path) have no rendered link to the corresponding `token_ledger` row at all.
                                       
                                        - ## 5. Admin FX-rate editor
                                       
                                        - `src/components/desk/billing/AdminPayfastPricingReview.tsx`, rendered only when `isAdmin` inside `Billing.tsx`. Reads/writes `admin_settings.payfast_usd_zar_rate` directly via the client, gated by RLS ("only platform admins can change this ... enforced by RLS on `admin_settings`"); the component's own copy states changes are audited. It shows the USD package list times the current/live rate to preview PayFast ZAR amounts, and a "Customer PayFast = VISIBLE/HIDDEN" and "Paystack public = true/false" status chip pair for at-a-glance review. If no rate is set, the panel shows "No rate set — PayFast customer checkout is blocked", matching `payfast-public-checkout.ts`'s `fx_rate_missing` rejection. This remains suitable unchanged if customer-facing ZAR wording elsewhere is hidden — it is already admin-only and does not need rework for the USD-first goal.
                                       
                                        - ## 6. Paystack visibility
                                       
                                        - Confirmed in `PaymentMethodPicker.tsx`: `PAYSTACK_PUBLIC_ENABLED = false` (module-level constant, with a comment explaining it is kept renderable for admins "so Paystack QA paths remain testable"). `showPaystack = PAYSTACK_PUBLIC_ENABLED || isAdmin`. For a normal customer this evaluates to `false`, so the Paystack button does not render at all on the `/desk/billing` picker. For an admin it renders with an explicit "[Admin only]" label and a non-customer-facing tooltip. Paystack code itself (webhook, verify, checkout-initiation) is untouched and was not read/modified in this pass. No Paystack customer-facing wording was found remaining for normal users on either billing surface.
                                       
                                        - ## 7. Data and audit requirements
                                       
                                        - Per the field-by-field analysis in Section 4, `token_purchases.metadata` and the `credits.purchase_initiated` audit row already retain the full requested set (USD package price, ZAR amount, FX rate, provider, provider reference, PayFast reference, package id, credit amount, mode) at checkout-initiation time. The gap is that the settlement-time `credits.purchased` audit row written by `payfast.ts` drops `price_usd`/`amount_usd`, drops the FX rate entirely, and uses different key names for the PayFast reference than what the one dedicated admin revenue view expects. This is a metadata-shape/consumer-mismatch gap, not a missing-storage gap — the underlying data is captured once (at initiation) but not re-attached at settlement, and the admin consumer was written against Paystack's USD-native field names.
                                       
                                        - ## 8. Settlement wording
                                       
                                        - Targeted search for "native settlement", "settles in ZAR", "Nedbank", "PayFast settlement", "ZAR settlement", "USD settlement" was run against `docs/` and `src/` only (not exhaustively against every generated evidence file). Occurrences found in this pass: `docs/payfast-primary-usd-pricing-fx-zar-report.md` (documentation only, not customer-facing). No occurrence of this wording was found rendered in any customer-facing component read in this pass (`Billing.tsx`, `BillingOverview.tsx`, `PaymentMethodPicker.tsx`, `PurchasesList.tsx`, `PayfastReturn.tsx`, `PayfastCancel.tsx`). The one concrete customer-facing settlement-adjacent wording that does need rework is the FX-rate/ZAR-amount disclosure in `PaymentMethodPicker.tsx` identified in Section 1 — it does not use the word "settlement" but has the same effect of exposing internal FX mechanics to the customer.
                                       
                                        - ## 9. Test coverage
                                       
                                        - Existing, confirmed-by-search coverage: `src/tests/payfast-phase-2j-customer-rollout.test.ts` (return/cancel read-only behaviour, public helper snapshots `usd_zar_rate`/`fx_rate_locked_at`/`amount_zar` into metadata), `src/tests/payfast-phase-2a-provider-identity.test.ts` (list-org-purchases selects `provider`/`provider_reference`), `src/tests/payfast-phase-2d-no-regression.test.ts` and `payfast-phase-2b/2c-no-regression.test.ts` (file-set regression guards covering `PaymentMethodPicker.tsx`, `PayfastReturn.tsx`, `PayfastCancel.tsx`). A no-external-FX-API guard test was not located in this pass and should be re-confirmed before Phase C.
                                       
                                        - ## 10. Recommended build phases
                                       
                                        - Phase A — Customer UI cleanup while keeping current PayFast ZAR processing. Files likely touched: `src/components/desk/billing/PaymentMethodPicker.tsx` (remove or admin-gate the ZAR amount/FX-rate disclosure line and button label; keep the underlying `computeDisplayZar`/`usdZarRate` plumbing intact for checkout math). Risk: low if only display strings change and no props/derivations feeding `payfast-public-checkout.ts` are touched. Tests required: a new test asserting the customer-facing picker never renders a ZAR amount or FX-rate string for a non-admin session. Must not change: `startPayfastPublicCheckout`, `buildPayfastPublicCheckout`, ITN handling, metadata snapshot fields.
                                       
                                        - Phase B — Admin/audit visibility hardening. Files likely touched: `src/components/admin/AdminRevenuePanel.tsx` (`purchaseFromAuditLog`/`purchaseFromLedger`: read `usd_zar_rate` in addition to `fx_rate`/`legacy_fx_rate`; read `provider_reference`/`pf_payment_id` as a fallback when `payment_reference`/`reference` are absent; consider carrying `price_usd` forward onto the settlement row, or joining back to the initiation row, so PayFast revenue counts in the native-USD totals instead of being forced to `$0`). Risk: medium — this changes what counts as "revenue" in existing dashboards and must be reconciled carefully against historical rows so totals do not silently shift without an accompanying explanation to whoever relies on that dashboard. Tests required: new coverage asserting a synthetic PayFast `credits.purchased` row surfaces non-zero USD revenue, a populated FX rate, and a populated reference in the panel's output. Must not change: PayFast ITN logic, Paystack webhook logic, `token_ledger`/ledger crediting.
                                       
                                        - Phase C — PayFast-provider capability branch, only after PayFast confirms USD/multi-currency support. Not started; explicitly out of scope until PayFast's own capability is confirmed directly with PayFast, per the brief's constraint against making USD-checkout assumptions.

                                              - ## Final status
                                             
                                              - PAYFAST_USD_FIRST_BILLING_AUDIT_COMPLETE_WITH_ADMIN_VISIBILITY_FOLLOW_UP
                                             
                                              - Reason: the requested payment metadata (USD price, ZAR amount, FX rate, provider, provider/PayFast reference, package id, credit amount, mode) is captured in `token_purchases.metadata` and the `credits.purchase_initiated` audit row at checkout time, but the one dedicated admin transaction view (`AdminRevenuePanel.tsx`) does not fully or correctly surface it for PayFast rows at settlement, due to (a) the settlement-time `credits.purchased` audit row omitting `price_usd`/`amount_usd` and the FX rate entirely, and (b) field-name mismatches between what PayFast writes (`usd_zar_rate`, `provider_reference`, `pf_payment_id`) and what the panel reads (`fx_rate`/`legacy_fx_rate`, `payment_reference`/`reference`). No code was changed in this pass.
