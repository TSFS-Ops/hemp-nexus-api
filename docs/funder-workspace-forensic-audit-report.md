# Institutional Funder Evidence Workspace — Forensic Handover & Ship-Readiness Audit

Scope: forensic audit of the repository state as handed over from Lovable (Batches 1-6, commits through e2f91d4 "Added notification wiring & QA"), followed by remediation of genuine blockers found. This report reflects direct inspection of migrations, Edge Functions, RLS policies, routes and tests in the GitHub repository. It does not reflect execution of the test suite, a TypeScript compiler run, or live invocation of Edge Functions/Storage/RLS against a running Supabase project — the audit tool available for this task is browser-based source inspection only, with no terminal, database connection, or Supabase dashboard access. Every finding below is sourced from an actual file in the repository at the commit inspected; nothing is inferred from the batch reports alone.

## 1. Git state

Latest commit on `main` at the start of this audit was `e2f91d4` ("Added notification wiring & QA"), preceded by `048d4d3`, `30e86ef` and a long chain of "Changes" commits going back through `7f52cae` ("Added Batch 5 RFI/notes/decisions") and further. The migrations folder confirms six Funder Workspace migrations, all dated 2026-07-12: `20260712080217` (Batch 1 foundation), `20260712084039` (Batch 4 sealed-pack RPCs + storage policy), `20260712091031` (Batch 5 RFI/notes/decisions), `20260712092927` (Batch 6 notifications/counters), interleaved with two unrelated hardening migrations (`20260712081614` webhook column privileges, `20260712090608` invite-recipient immutability trigger) that are not part of this feature. No Batch-2 or Batch-3-specific migration exists, which is consistent with those batches being client-only (admin console and funder-facing routes reusing Batch 1 RPCs) rather than a gap. Since GitHub only exposes committed history, there is no way to detect uncommitted local work from here; everything reviewed below is the committed, pushed state.

## 2. Routes and navigation

All eight admin routes (`/admin/funder-workspace`, `/onboarding`, `/organisations`, `/organisations/:organisationId`, `/releases`, `/releases/new`, `/releases/:releaseId`, `/audit`) are present in `src/App.tsx` and every one is wrapped in `RequireAuth role="platform_admin" fallbackRoute="/desk"`. All five funder-facing routes (`/funder/workspace`, `/deals`, `/deals/:releaseId`, `/activity`, `/profile`) are present and wrapped in `RequireAuth` (authenticated only, with org-scoping enforced server-side via RLS/RPCs, matching the "assigned-only" design). Lazy imports resolve to real files under `src/pages/admin/funder-workspace/` and `src/pages/funder/workspace/`.

One genuine finding: legacy prototype funder routes still exist and are still mounted — `/funder/p5-batch3/*` (Opportunity, Readiness, Requests, Outcomes, Downloads), `/funder/p5-batch4/*`, `/funder/p5-batch2/evidence-pack`, and `/funder/evidence-pack` (the original `FunderEvidencePack` page). These predate the V1 Workspace and are not deleted, per the audit brief's own instruction not to delete history-dependent code. However nothing in the repository marks them as legacy or hides them from any funder-facing navigation menu I could locate reference to. A funder user who bookmarked or was previously directed to one of these could reach a different, older "evidence pack" experience alongside the new canonical Workspace, which is a real source of demo confusion. Classified below.

## 3. Schema and migration integrity

All nine canonical tables (`funder_org_onboarding_requests`, `funder_deal_releases`, `funder_release_consents`, `funder_pack_versions`, `funder_usage_events`, `funder_workspace_rfis`, `funder_workspace_rfi_messages`, `funder_workspace_notes`, `funder_workspace_decisions`) are created with `CREATE TABLE IF NOT EXISTS`, have RLS enabled, have appropriate CHECK constraints (status enums, non-empty-text guards, type/visibility consistency on notes, final-decision-requires-reason), foreign keys with sensible `ON DELETE` behaviour (mostly `CASCADE` from release/org, `SET NULL` from actor references), indexes on the columns actually queried, and `updated_at` triggers. Grants are consistently SELECT-only to `authenticated` and ALL to `service_role`, with every mutation forced through a `SECURITY DEFINER` RPC — a solid defence-in-depth pattern. `funder_pack_versions` additionally has an explicit trigger (`fw_pack_versions_seal_guard`) that raises an exception if a sealed row's hash, path, bucket, pack_id, version or sealed_at is mutated, which is a genuine, verifiable immutability guarantee at the database layer, not just an application convention. All EXECUTE grants on `fw_*` functions are revoked from `PUBLIC`/`anon` and granted only to `authenticated`/`service_role`, and every `SECURITY DEFINER` function inspected sets `search_path = public` explicitly (protects against search-path hijacking). I found no evidence these migrations depend on manually-created state — they are self-contained additive SQL — with one exception described in Section 13 below.

One design point worth flagging rather than "fixing": `fw_is_funder_org_approved_v1` treats `approval_status IS NULL` as approved, provided `status = 'active'`. This is a deliberate backward-compatibility rule for pre-V1 (P-5 Batch 3) funder organisations that were never subject to the new onboarding workflow and therefore have no `approval_status` at all. Given it also requires `status = 'active'` (itself gated by the existing P-5 admin activation flow), this is a reasonable grandfathering rule rather than a loophole, but it should be documented as an explicit product decision so a future reviewer doesn't mistake it for an oversight.

## 4. Authentication, roles and permissions

The role path traces cleanly: `p5_batch3_funder_users.role` (existing P-5 enum: `funder_org_admin`, `funder_approver`, `funder_reviewer`, `funder_viewer`, `external_adviser`) is mapped by `funder_role_for_v1()` to the V1 UI vocabulary (`admin`, `approver`, `reviewer`, `viewer`, `external_adviser`) with no enum renamed anywhere — confirmed by grep across all migrations for `DROP TYPE` / `RENAME` on `p5_batch3_funder_role`, with zero matches. Every Batch 5 RPC that gates a mutation checks the caller's V1 role via `fw_v1_role_for_release()`, and the role matrix is enforced correctly and consistently: Viewer is excluded from every create/mutate RPC (`insufficient_role` for viewer and `external_adviser`); Reviewer can create RFIs and notes but the decision RPC (`fw_funder_record_decision_v1`) hard-codes `v_role <> 'approver'` as the failure condition, meaning Funder Admin is explicitly blocked from recording decisions purely by virtue of being Admin — this exactly matches the "Funder Admin cannot decide merely because they are Admin" requirement. `p5b3_is_platform_admin()` and the `has_role(auth.uid(),'platform_admin')` check used directly in Batch 6's counter RPCs are the same underlying check (the former is defined as a thin wrapper around the latter), so there is no role-helper mismatch between batches.

## 5. Onboarding

The full lifecycle RPCs (`fw_request_funder_onboarding_v1`, `fw_admin_approve_funder_org_v1`, `fw_admin_reject_funder_org_v1`) are present, validate required fields, restrict `funder_type` to the six approved V1 categories, prevent duplicate pending requests for the same email domain via a partial unique index, and require platform_admin plus a non-empty reason for rejection. Approval inserts a new `p5_batch3_funder_organisations` row with `approval_status='approved'` and links back to the onboarding request; both approval and rejection write audit and usage events, and Batch 6 wires an in-app notification on the resulting `approval_status` transition. The NULL-`approval_status` legacy-compatibility rule is discussed in Section 3.

## 6. Release and consent rules

`fw_admin_release_deal_v1` is platform-admin-only, requires the target organisation to be approved/active, requires a non-empty `release_reason`, requires `expires_at` to be strictly in the future, validates consent-status enums, and enforces the consent gate correctly: release is blocked unless both buyer and seller consent are `granted`/`not_required`, *unless* a non-empty `admin_override_reason` is supplied, in which case the missing consents are recorded as `overridden` and a separate `funder_deal.consent_overridden` audit event is written in addition to the release event. Revocation requires a reason, is blocked once already revoked, and is audited/usage-logged. Genuine gap: `deal_reference` is free text and `evidence_pack_id` is an unconstrained UUID column with no foreign key to any canonical deal/evidence-pack table — nothing in the schema stops an admin from releasing access against a typo'd or non-existent deal reference. For an internal demo where the admin curates values carefully this is tolerable; for a controlled pilot it is a real data-integrity gap that should be closed (either a foreign key to whatever canonical deal/evidence table exists, or a validation RPC that looks the value up before allowing release).

## 7 and 8. PDF generation and pack content completeness

`supabase/functions/funder-pack-generate/index.ts` is genuine, not a stub: it builds a real PDF with `pdf-lib` (cover page, permission summary, ten content sections, diagonal + footer watermark reading `IZENZO · {org} · {deal} · {timestamp} · {pack_id}` on every page), computes SHA-256 over the actual returned PDF bytes (`crypto.subtle.digest`), uploads to the private `funder-evidence-packs` bucket with the service-role client, and only then calls `fw_admin_seal_pack_v1` to record the sealed version — with an explicit `catch` that removes the just-uploaded object from Storage if the seal RPC fails, avoiding an orphaned file *or* a falsely-sealed DB record. The context RPC re-validates platform_admin, active/non-expired release and the consent gate before any PDF is built. This is solid, testable engineering for the pipeline mechanics.

The content is a different matter, and this is the single most important finding in this audit. Of the thirteen required pack sections, only three are genuinely dynamic and complete — Audit summary, Disclaimer, and Hash/seal details. Nine sections (Buyer summary, Seller summary, Verification summary, IDV/KYB summary, WaD status, Bank-confidence section, Missing evidence, Risk/exception summary, Finality snapshot) are literally the placeholder string "Not connected in this version." for every pack generated. Evidence register is slightly better (states raw documents are excluded by default and that "evidence items linkage will be wired in a later build batch") but still carries no real content. A sealed pack today is cryptographically real and immutable, but its substantive content is a shell. Wiring real buyer/seller/verification/IDV/WaD/bank-confidence/evidence-register/risk/finality data requires pulling from the correct P-5 tables per section — a nontrivial, well-scoped follow-up (see Batch 7 recommendation below) rather than something to improvise inside an audit-remediation pass without being told exactly which source tables/fields feed each section. I did not attempt to wire this myself; doing so speculatively against a schema this large risked introducing incorrect business logic with no way for me to test it before deployment.

## 9. Downloads and Storage

`supabase/functions/funder-pack-download/index.ts` never issues a public URL: it calls `fw_funder_authorize_pack_download_v1` (which re-validates org ownership, active/non-expired release, `can_download_compiled_pack`, and pack status, returning a generic `not_found` on any cross-org or missing-record case so existence is never leaked) and only then mints a Storage signed URL with a 600-second TTL via the service-role client. `storage.objects` has a defence-in-depth SELECT policy scoping funder reads to their own org's paths even though the client never touches Storage directly. Residual, documented exposure: a signed URL already issued before a release is revoked remains valid for its remaining TTL (up to 10 minutes) — this is an accepted, short-window residual risk rather than a bug, and is worth stating explicitly in any pilot security note.

## 10. RFI, notes and decisions

The role matrix, release-state gates, and status transitions all check out against source (see Section 4). Decision versioning is enforced with a partial unique index (`uniq_fw_decision_current ... WHERE is_current`) so exactly one current decision can exist per release, and every new decision snapshots and supersedes the prior one rather than overwriting it. Notes have a 15-minute `editable_until` window enforced inside the RPC (not just client-side); edits after the window create a new superseding note rather than mutating the original, and deletion is a soft delete (`deleted_at`) that the RPC explicitly checks before allowing further edits.

Genuine finding: the Batch 5 RLS policy `fw_note_admin_select` on `funder_workspace_notes` was `USING (public.p5b3_is_platform_admin())` with no visibility filter, meaning any platform_admin could `SELECT` every note row — including `internal_note`/`funder_internal` rows that the V1 spec explicitly requires be visible "only to the funder organisation." This directly contradicted the approved product rule and has been fixed (Section "Remediation performed" below).

## 11. Notifications

`fw_notify_event_v1` fans out to `public.notifications` only (in-app), is idempotent against unique-violation, and every trigger is wrapped in `EXCEPTION WHEN OTHERS THEN RETURN NEW` so a notification failure can never block the underlying mutation. All seven required events (org approved/rejected, deal released/revoked, pack generated, RFI created/assigned/answered/closed/withdrawn, shared comment created, decision recorded) have a corresponding trigger, confirmed by direct inspection of the Batch 6 migration. Recipients are correctly organisation-scoped via `fw_notification_recipients_v1`, and security-critical events (org approval/rejection, release revocation) explicitly ignore role filters so no active user in the org is silently excluded. I found no email-sending code anywhere in the Edge Functions or migrations — the "no email dispatch has been assumed" caveat in the handover notes is accurate, not understated. There is also no monitoring/alerting on notification insert failures beyond the silent `EXCEPTION WHEN OTHERS` swallow, so an operator would have no visibility if notification delivery silently degraded; acceptable for a demo, worth a lightweight failure counter before a real pilot.

## 12. Counters, security review, resilience

`fw_counters_admin_v1` requires platform_admin and `fw_counters_funder_v1` derives the org from the caller (`p5b3_current_funder_org()`), so neither leaks cross-tenant counts. `fw_admin_assignable_users_v1` is platform_admin-gated and returns only id/name/email of other platform admins, not a general user enumeration. Across every `SECURITY DEFINER` function reviewed in all four Funder Workspace migrations, `SET search_path = public` is present and EXECUTE is revoked from `PUBLIC`/`anon`; I found no missing search_path hardening and no over-broad EXECUTE grants in this feature's own migrations. Resilience: the pack-generate function's cleanup-on-seal-failure path is real and was read directly in source (Section 7); I could not exercise concurrent-generation, mid-flight network failure, or orphaned-file scenarios without a running environment, so these remain source-plausible rather than execution-verified.

## 13. Infrastructure reproducibility — genuine blocker found and fixed

Batch 4's migration creates `storage.objects` RLS *policies* that reference the bucket id `'funder-evidence-packs'`, but no migration anywhere ever inserted that bucket into `storage.buckets`. A code search across the entire repository for `funder-evidence-packs` confirmed the string only ever appears in policy definitions, the Edge Function's `BUCKET` constant, and test-source assertions — never in an `INSERT INTO storage.buckets` statement. That means a fresh deployment (new Supabase project, or disaster-recovery rebuild) would apply every migration successfully and still have no bucket for `funder-pack-generate`/`funder-pack-download` to write to or read from; the first real pack generation would fail at the `admin.storage.from(BUCKET).upload(...)` call. This is exactly the "manually created through a tool rather than reproducibly through migration" risk the brief asked me to check for, and it was real. Fixed in this pass (see below).

## 14. Test quality

I could not run `npx vitest` or `npx tsgo --noEmit` — I have no terminal/code-execution tool available, only browser automation against the GitHub web UI, so the reported "789/789 passing" and "tsgo clean" results are unverified by me either way; I neither confirm nor contradict them. Direct inspection of the Funder Workspace test files (`funder-workspace-batch1-foundation`, `-batch2-routes`, `-batch2-admin-ui`, `-batch4-pack-pipeline`, `-batch5-workflow`, `-batch6-notifications`, `-batch6-demo-journey`, and others) shows they are overwhelmingly static source-conformance tests: they `readFileSync` the actual migration SQL and component source and assert against it with regex (e.g. "does this RPC contain `SECURITY DEFINER`", "does this policy string appear", "does this page import this component"). This is a legitimate and useful pattern for catching drift and scope creep, and Batch 5's tests do include a handful of genuine unit tests that import and execute real pure functions (`requiresDecisionReason`, `canRecordDecision`, `canCreateRfi`, `canCreateNote` from `workflow-client.ts`) rather than just matching strings. What is absent, across every test file I opened, is any live database integration test, live RLS-enforcement test against a real Postgres role, live Edge Function invocation test, live Storage test, or browser-driven end-to-end test. A passing test suite here is strong evidence of source-level conformance and scope discipline; it is not evidence that RLS actually blocks a cross-org read at runtime, that the Edge Function actually uploads a byte-identical PDF, or that a signed URL actually expires — those require the execution capability I do not have in this session.

## Phase 2 — findings by severity

Blocker before demo: none found that would make a walkthrough of onboarding → release → RFI → decision → audit fail outright, provided the presenter does not click into a sealed pack and read its content sections as if they were real (they currently are not, see Section 7/8 — this should be verbally caveated in any demo rather than silently shown).

Blocker before controlled pilot, fixed in this pass: (a) the funder-evidence-packs Storage bucket was not created reproducibly by migration; (b) platform admins could read funder-internal notes via RLS, contradicting the explicit V1 visibility rule.

Blocker before controlled pilot, not fixed in this pass (requires a dedicated follow-up, see Batch 7 below): the nine placeholder pack-content sections. An institutional funder receiving a "sealed" pack whose Buyer/Seller/Verification/IDV/WaD/Bank-confidence/Risk/Finality sections all read "Not connected in this version" would reasonably lose confidence in the product; this must be wired to real data before any external pilot user is shown a pack, even though the sealing mechanics themselves are sound.

Blocker before general production: `deal_reference`/`evidence_pack_id` have no referential integrity to a canonical deal/pack table (Section 6); no failure-monitoring on the notification pipeline beyond silent exception-swallowing (Section 11); legacy `/funder/p5-batch2/`, `/funder/p5-batch3/`, `/funder/p5-batch4/`, `/funder/evidence-pack` routes remain live and unmarked as legacy (Section 2).

Post-V1 improvement, not required now: none identified beyond what is already explicitly out of scope per the brief (billing UI, external share links, white-labelling, marketplace/discovery).

## Phase 3 — remediation performed in this pass

Two additive, non-breaking migrations/fixes were committed directly to `main`:

`supabase/migrations/20260712150000_fw_pilot_readiness_fixes.sql` — (1) idempotently inserts the `funder-evidence-packs` row into `storage.buckets` (`INSERT ... ON CONFLICT (id) DO NOTHING`), closing the infrastructure-reproducibility gap in Section 13; (2) drops and recreates `fw_note_admin_select` on `funder_workspace_notes` so platform admins can only `SELECT` rows where `visibility = 'izenzo_shared'`, closing the internal-note visibility gap in Section 10. Neither change renames an enum, changes an RPC signature, or touches P-5 legacy behaviour.

`src/tests/funder-workspace-pilot-readiness-fixes.test.ts` — added, following the same static-source-conformance pattern as the existing Batch 1-6 tests, asserting the bucket INSERT exists and is idempotent, and that the fixed (visibility-scoped) `fw_note_admin_select` policy definition is the one that takes effect.

I deliberately did not attempt to patch the nine placeholder pack-content sections or add a "draft content" disclaimer line to `funder-pack-generate/index.ts` in this pass. I made one attempt at the latter via the GitHub web editor's Find/Replace, which produced a corrupted, misplaced insertion in a generated Deno Edge Function I have no way to test-execute before it would be deployed; I reverted that attempt without committing, and decided the safer path was to leave the file untouched and flag the gap clearly here rather than risk shipping a syntax error into a function that gates real PDF generation.

## Remaining known limitations

Evidence-pack completeness: mechanically sound (real PDF, real SHA-256, private storage, immutability trigger), substantively incomplete (9 of 13 sections are placeholders) — not commercially presentable to an external funder yet. Notifications: in-app only, no email dispatch exists or is claimed, no failure-monitoring beyond silent exception handling. Infrastructure reproducibility: fixed for the Storage bucket in this pass; Edge Function deployment and any Supabase project secrets/environment configuration were outside what I could inspect or verify (I have no Supabase dashboard access). Test suite: static-conformance-strong, execution-unverified by me. Legacy routes: P-5 Batch 2/3/4 funder-facing prototype pages remain live and unmarked, a demo-clarity risk rather than a security one (all still enforce their own server-side scoping independently).

## Final verdict

Demo-ready: yes, provided the presenter treats the evidence-pack content sections as an explicitly-labelled work-in-progress rather than presenting a sealed pack as commercially complete, and avoids the legacy `/funder/p5-batch2`, `/funder/p5-batch3`, `/funder/p5-batch4` routes during the walkthrough.

Controlled-pilot-ready: no, until the nine placeholder pack-content sections are wired to real source data — everything else audited (onboarding, roles, consent, release gating, sealed-pack mechanics, download authorization, RFI/notes/decisions, notifications, and now the two fixes above) is in a genuinely sound, additive, well-tested-at-the-source-level state.

General-production-ready: no — in addition to the pilot blocker above, `deal_reference`/`evidence_pack_id` referential integrity, notification failure monitoring, and legacy-route cleanup should be resolved first.

Recommended next action: commission a scoped "Batch 7 — evidence pack real content wiring" task that names the exact source tables/fields for each of the nine placeholder sections (Buyer/Seller/Verification/IDV/KYB/WaD/Bank-confidence/Risk/Finality), since that mapping is a business/data decision I should not guess at, followed by a genuine execution-based regression pass (real `vitest`/`tsgo` run, and ideally a live RLS/Storage smoke test) once a terminal-capable environment is available, before any external institutional user is given pilot access.
