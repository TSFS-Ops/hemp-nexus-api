# PayFast Settlement-to-Bank Tracking - Build Plan - 2026-07-23

Repository: TSFS-Ops/hemp-nexus-api. Proceeds from docs/payfast-enterprise-readiness-audit-2026-07-23.md, which identified the absence of PayFast settlement-to-bank tracking as the main remaining enterprise gap, alongside a secondary gap of no consolidated operator runbook.

Scope of this task: planning and design only. No code was changed, no PR was opened, nothing was merged. PR #28, PR #29 and PR #30 were not touched. PayFast checkout, PayFast ITN verification, wallet and ledger crediting, and Paystack were not modified. No historical rows were mutated. No FX logic was introduced. No external banking integration was added; this plan explicitly recommends against building bank scraping or a live bank/PayFast payout API integration in the first phase.

## 1. Problem statement

A PayFast payment can succeed and the Izenzo wallet can be credited correctly, while the application still has no way of knowing whether PayFast has actually settled that money into Izenzo's own bank account. Enterprise-grade payment operations need to distinguish several distinct moments in a purchase's life: checkout created, customer paid, PayFast ITN confirmed, wallet credited, PayFast settlement expected, bank settlement confirmed, settlement delayed, and settlement exception requiring manual review. Per the prior audit, the first four of these are already solid and are not touched by this plan. This document designs the smallest, safest addition that closes the remaining four states without touching any existing payment-critical code path.

## 2. Recommended data model

The recommended object is a new table, payment_settlements, kept deliberately separate from token_purchases and token_ledger so that settlement tracking is purely additive and can never affect crediting correctness or the ledger's hash-chain style guarantees. The schema is written provider-generically, with a provider column, so it could in principle cover Paystack later, but only PayFast rows would be populated in this first build.

Recommended fields: id as the primary key; provider, constrained to payfast for now; provider_reference, matching the value already stored on token_purchases; token_purchase_id, a foreign key to token_purchases.id with a unique constraint so at most one settlement row can ever exist per purchase; org_id, denormalised from the purchase for convenient RLS and querying; amount_zar and amount_usd, copied from the purchase at creation time rather than recomputed; usd_zar_rate, also copied for display, not recalculated; expected_settlement_date, a date computed from the ITN-confirmed timestamp plus a configurable number of business days; settlement_status, discussed in section 3; settlement_confirmed_at and settlement_confirmed_by, populated only when an admin marks a row confirmed; bank_reference, a free-text field intended to hold an internal PayFast payout batch id or a short manual reconciliation note, never a real bank account or routing number; notes, free text for admin annotations; exception_reason, free text populated when status moves to exception; created_at and updated_at timestamps; and a metadata jsonb column to carry audit-relevant context such as which process created the row, mirroring the pattern already used elsewhere in this codebase for admin_risk_items. All monetary fields are copied at creation time rather than joined live, so that later changes to a purchase row, which should not happen anyway, cannot silently alter historical settlement figures.

## 3. Recommended status model

The brief lists eight candidate statuses: not_applicable, expected, pending, confirmed, delayed, exception, reconciled, and cancelled. Building all eight would overbuild a first version, since several overlap or require a real bank feed that does not exist yet. The recommended minimal set is five statuses: expected, confirmed, delayed, exception, and cancelled.

expected is the default status written when a settlement row is first created, meaning PayFast is expected to settle this payment into the bank in due course; it replaces the need for a separate pending status, since before confirmation there is only one waiting state worth distinguishing. confirmed means an admin has verified, today by manually checking a bank statement or the PayFast merchant dashboard, that the money has arrived, and requires settlement_confirmed_at and settlement_confirmed_by to be set. delayed means the expected settlement date has passed without confirmation; this is designed to be set automatically by a scheduled job rather than only by an admin, so it behaves as a genuine operational signal rather than a manual label. exception means something is wrong, such as a suspected amount mismatch or a payment that needs manual investigation, and requires an exception_reason. cancelled covers the case where the underlying purchase is later refunded before settlement was confirmed, so the settlement row can be closed out without pretending money is still expected.

not_applicable is not recommended as a stored status, because settlement rows are only ever created for purchases that qualify in the first place; if a purchase does not qualify, per section 4, no row is created, so there is nothing to mark not-applicable. pending is intentionally folded into expected, since maintaining a separate status without a real bank feed to distinguish not yet due from overdue and still not confirmed adds complexity without operational value; delayed already covers the overdue case. reconciled is deferred rather than dropped entirely: today, confirmed already functions as the closest thing to reconciliation the business can do, given there is no live bank statement feed to reconcile against. If Izenzo later gains an automated bank statement import, reconciled can be added at that point as a genuinely distinct state meaning matched programmatically against a real statement line, which is different from an admin's manual confirmation today.

## 4. How settlement records are created

A settlement record should be created only for a PayFast purchase that has actually completed successfully, meaning the ITN has been verified and the wallet has already been credited. No row should ever be created for a checkout that was only initiated, for an ITN that failed verification, or for a purchase that was cancelled before payment. Exactly one settlement row should exist per completed PayFast purchase, enforced at the database level by the unique constraint on token_purchase_id described in section 2, so that duplicate ITN deliveries, which the existing crediting path already deduplicates, cannot also create duplicate settlement rows.

Two implementation approaches were considered for triggering creation. The first is to insert the settlement row inline, at the end of the existing PayFast ITN success path, immediately after crediting succeeds. This gives the lowest latency, since the row appears the instant the payment is confirmed, but it requires touching the ITN function's source file, which sits close to the payment-authenticity and crediting logic this task was explicitly told not to change. The second approach is a separate, small, idempotent reconciliation job, following the same pattern already used by this codebase's existing side-effect-reconciliation function, that periodically scans completed PayFast purchases and inserts a settlement row for any that do not yet have one. This is the recommended approach for the first build, because it never touches the ITN function or the crediting function at all, it reuses a proven idempotent pattern already in the codebase, and it has a useful side effect: if the job ever fails to create a row for a completed purchase within a reasonable window, that gap is itself the natural trigger for the "PayFast paid but no settlement record" alert described in section 8. Inline creation can be revisited later, as a small, separate, and separately reviewed follow-up, once the job-based approach has proven itself in production, if lower latency turns out to matter operationally. No automatic bank confirmation should ever be written by either approach; moving a row to confirmed is always an explicit admin action, since no real settlement feed exists to automate it safely.

## 5. Admin UI

The recommended surface is a new, dedicated component, AdminPayfastSettlementsPanel.tsx, rather than extending the existing AdminRevenuePanel.tsx. Settlement tracking is an operationally distinct concern from revenue reporting, and the existing revenue panel already has a specific, tested job; adding a new panel keeps both simpler and keeps this change lower-risk, while the two can still cross-link using the shared provider_reference. The new panel would be wired into the HQ admin shell alongside the other existing Admin panel components, gated the same way they are.

The panel should show, per settlement row, the PayFast provider reference, the customer or organisation name, the USD price, the ZAR amount, the FX rate used at checkout, the date the PayFast ITN was confirmed, the date the wallet was credited, the expected settlement date, the current settlement status as a badge, the bank reference if one has been recorded, the exception reason if any, and any admin notes. Admin actions on each row should allow marking the settlement confirmed, marking it delayed, marking it as an exception, adding or editing a bank reference, and adding a note; a CSV export of the visible rows should be available for offline reconciliation work, and each row should offer a way to view its audit trail, meaning the history of status changes and who made them. None of this panel, or the underlying table, should ever be reachable by a normal customer; it is an admin-only surface end to end.

## 6. Roles and permissions

Writes to payment_settlements, meaning marking a row confirmed, delayed, or an exception, or editing the bank reference or notes, should be restricted to the platform_admin role, using the same has_role SECURITY DEFINER RPC check already used throughout this codebase's admin-only edge functions, such as the existing PayFast sandbox checkout gate and the refund approval functions. Reads could reasonably be extended to a read-only role such as compliance_analyst, which this codebase already uses for governance-record visibility elsewhere, but this should be confirmed against the current approved role registry in docs/role-negative-e2e-coverage.md before implementation rather than assumed. No raw bank account numbers, routing numbers, or other sensitive banking identifiers should ever be accepted into the bank_reference or notes fields; these fields are intended only for internal reference strings such as a PayFast payout batch id or a short reconciliation comment. All reads and writes should go through a governed edge function using the service-role client, the same pattern used by list-org-purchases and the refund-approval functions, rather than allowing any direct client-side Supabase table access, and row-level security on the table itself should default-deny all client roles so the edge function is the only path in.

## 7. Audit behaviour

Every write to a payment_settlements row, meaning any status change, any bank_reference edit, or any note added, should generate an audit_logs row recording the actor, the action, and the before and after state, following the same governance pattern already used by the refund approval flow. Audit rows should never be edited or deleted after the fact, only appended to, consistent with how audit_logs already behaves elsewhere in this codebase. The admin UI's audit trail view described in section 5 is simply a read of this same audit_logs history, filtered to the relevant settlement row, rather than a second parallel history mechanism.

## 8. Alerts and exception rules

The recommended approach is to extend the existing admin_risk_items exception queue, described in the prior audit, with a small number of new PayFast-settlement-specific kinds, rather than building a separate alerting mechanism. payfast_settlement_overdue would be raised, at medium severity, when a settlement row's expected_settlement_date has passed by more than a configurable number of business days while it is still in the expected status; this is the same condition, and could be the same job, that flips the status to delayed. payfast_settlement_exception would be raised, at high severity, whenever an admin marks a row as exception, capturing the exception_reason. payfast_paid_no_settlement_record would be raised, at high severity, if a completed live PayFast purchase has no matching settlement row after a grace period, which primarily guards against the reconciliation job itself failing or falling behind. payfast_settlement_amount_mismatch is designed into the schema and the alert taxonomy now, but is intentionally deferred in practice until a real bank or PayFast settlement feed exists to compare against, since without one there is nothing to mismatch against beyond the admin's own manual entry. A softer, UI-level validation, rather than a new alert kind, is recommended for the case where policy requires a bank_reference before a row can be marked confirmed; this is simpler to enforce as a form validation than as a separate monitored exception. All new alert kinds should follow the existing dedup_key convention already used by admin_risk_items, so that repeated checks of the same underlying condition update one open row rather than creating alert noise, exactly as the existing payfast_itn_rejected and missing_side_effect kinds already do.

## 9. Tests needed before implementation

A settlement row should be proven to be created after a valid PayFast completion, and no row should be created for an invalid or rejected ITN. A duplicate ITN delivery for the same purchase should be proven not to create a second settlement row, exercising the unique constraint on token_purchase_id. An admin holding the platform_admin role should be proven able to mark a row confirmed, delayed, or an exception, and a non-admin, including a normal customer and any other role, should be proven unable to perform any of those actions. Every status change should be proven to write an audit_logs row. A settlement row left in the expected status past its expected_settlement_date should be proven to surface in the exception and alert queue described in section 8. Settlement rows and the panel that lists them should be proven never to be visible to a customer-facing query or route, mirroring the existing tenant-isolation tests for purchase history. Older, already-completed PayFast purchases that have no settlement row yet, because this feature did not exist when they were made, should be proven not to break AdminRevenuePanel or any other existing admin surface; those surfaces should degrade gracefully by simply showing no settlement data for such rows rather than erroring. Finally, a no-regression test should assert that PayFast checkout, PayFast ITN verification, and wallet and ledger crediting are all byte-for-byte unaffected by this feature's addition, following the same no-regression pattern already used for prior PayFast phases.

## 10. Migration and backfill strategy

The recommended default is not to backfill historical PayFast purchases automatically as part of the first build. The first build should only create settlement rows prospectively, for purchases completed from the moment the reconciliation job goes live onward. This keeps the initial change small, keeps its blast radius limited to new data only, and avoids any risk of a bulk migration touching historical token_purchases rows, which this task was explicitly told not to mutate; note that a backfill of this kind would only ever insert new payment_settlements rows and would never need to touch token_purchases or token_ledger at all, but it is still safer to defer until the core feature is proven.

A separate, optional, admin-controlled backfill can be offered later as its own small follow-up, once the prospective creation and admin UI have been live and stable for some time. That backfill would be triggered explicitly by an admin action, would only insert payment_settlements rows for historical completed PayFast purchases that do not yet have one, defaulting their status to expected with an expected_settlement_date computed the same way as for new rows, and would never modify token_purchases or token_ledger. It should itself be idempotent and safe to run more than once.

## 11. Risks

The main risk is scope creep into a full bank integration; this plan deliberately avoids that, and any future move toward ingesting a real PayFast payout report or bank statement feed should be treated as a separate, larger, and separately scoped project with its own review, not folded into this build. A second risk is accidentally touching the PayFast ITN function while wiring up settlement-row creation; this is why the job-based creation approach in section 4 is recommended over inline creation. A third risk is exposing real bank details through the bank_reference or notes fields if they are ever misused; this is mitigated by treating those fields as opaque internal reference strings only, documented plainly in the schema and the admin UI, and never accepting anything that looks like an account or routing number. A fourth risk is exposing settlement data to the wrong audience, either customers or an unintended role; this is mitigated by the default-deny row-level security and governed edge-function pattern in section 6, and should be proven by the tenant-isolation and role tests in section 9. A fifth risk is treating this generic, provider-tagged schema as an invitation to also populate Paystack settlement rows without a separate design pass; this plan only recommends populating PayFast rows in the first build, and Paystack support, if ever wanted, should get its own short design review given Paystack's different, already admin-only, legacy status.

## 12. Implementation phases and recommended branch names

Phase 1 is the schema: a migration creating payment_settlements with the fields from section 2, the unique constraint on token_purchase_id, default-deny row-level security, and supporting indexes. Recommended branch: feature/payfast-settlement-schema-2026-07-23. This phase carries low risk, since it only adds a new table and touches nothing existing.

Phase 2 is the reconciliation job: a small, idempotent scheduled function that creates settlement rows for completed PayFast purchases missing one, following the side-effect-reconciliation pattern, plus the payfast_paid_no_settlement_record and payfast_settlement_overdue admin_risk_items wiring from section 8. Recommended branch: feature/payfast-settlement-reconciliation-job-2026-07-23. Tests for this phase should be bundled in the same branch, following this codebase's existing convention of pairing each phase with its own no-regression test file, for example payfast-settlement-phase-2-no-regression.test.ts.

Phase 3 is the admin surface: AdminPayfastSettlementsPanel.tsx plus a governed edge function, for example admin-payfast-settlement-update, implementing the confirm, delay, exception, bank-reference, and note actions from section 5, with the role gating from section 6 and the audit writes from section 7. Recommended branch: feature/payfast-settlement-admin-ui-2026-07-23.

Phase 4 is documentation: updating or creating the consolidated PayFast operator runbook identified as the secondary gap in the prior audit, adding a settlement-reconciliation procedure once phases 1 through 3 are live. Recommended branch: docs/payfast-operational-runbook, which can proceed in parallel with phases 1 through 3 rather than waiting for them.

Phase 5 is the optional historical backfill described in section 10, deliberately sequenced last and only undertaken once phases 1 through 3 have been live and stable for some time. Recommended branch: feature/payfast-settlement-backfill-2026-07-23. This phase is explicitly optional and future-facing rather than part of the initial build.

## 13. Open questions and items blocked on further information

None of these block the plan itself from being ready, but each should be resolved during Phase 1 through Phase 3 rather than assumed. First, whether PayFast exposes any settlement or payout API or report that Izenzo could ingest was not found evidenced anywhere in this repository; without one, confirmed will remain a manual admin claim rather than an automated match, and this should be confirmed with PayFast or with Izenzo's own Nedbank access before assuming otherwise. Second, the exact number of business days to use for expected_settlement_date and for the delayed threshold is a business decision, not a technical one, and should be confirmed with Izenzo operations rather than guessed; this plan does not assume a specific number. Third, whether reads of the new panel should be extended to compliance_analyst or kept to platform_admin only should be confirmed against the current approved role registry before Phase 3. Fourth, any expected format for bank_reference, for example a specific PayFast payout batch id format, is blocked on seeing a real PayFast settlement report or statement, and should be treated as free text until that is available.
