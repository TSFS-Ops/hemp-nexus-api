## [4.2.0] - 2026-05-03

### USD-Native Billing Cutover (FX layer retired)

- **[Billing]** Paystack confirmed settling **directly in USD** as of 2026-05-01. Removed FX layer (exchangerate.host + cached fallback) for new purchases.
- **[Billing]** 1 credit = $1.00 USD. Tier prices: `single` $1, `pack_10` $10, `pack_50` $45 (-10%), `pack_200` $160 (-20%).
- **[Billing]** New purchases write `price_usd` + `currency='USD'` + `fx_basis='native_usd'`. Pre-cutover ZAR rows preserved for historical reporting.
- **[Docs]** Customer-facing docs (`public/docs/live-test-script.md`, `public/docs/getting-started.md`, `docs/api-reference.md`, `docs/how-to-test.md`, `docs/core-positioning.md`) updated to USD pricing.
- **[CI]** New `scripts/check-docs-no-zar-billing.mjs` lint runs in `prebuild` and fails CI if customer-facing ZAR billing strings reappear in `public/docs/`.

## [4.1.0] - 2026-04-27

### POI Pre-Verification Policy (binding clarification)

- **[Compliance]** Pre-POI light intel must be **system-assisted** — auto-generated by `counterparty-intel-auto` edge function via Lovable AI Gateway. Never user-typed. NO hard verification before POI; NO paid APIs.
- **[UI]** `CounterpartyIntelPanel` is read-only with auto-run + Refresh, no input fields.
- **[Schema]** `match_counterparty_intel` table gains `auto_summary` / `auto_sources` / `auto_status` columns.
- **[Admin]** Optional `OperatorVerificationClipOn` admin clip-on never blocks POI.
- **[Compliance]** WaD 9-gate hard verification unchanged.

## [4.0.0] - 2026-04-22

### SECDEF Stage D1 — Database Privilege Lockdown

- **[Security]** `service_role`-only `EXECUTE` on critical atomic functions: `atomic_token_burn`, `atomic_generate_poi_v2`, `atomic_accept_bind`, `atomic_engagement_transition`, `atomic_validate_governance_doc`, `is_test_mode_bypass_enabled`, `is_production_environment`. Never re-grant to `authenticated` / `anon` / `PUBLIC`.
- **[Architecture]** All atomic functions must be invoked through their owning edge function — never directly from the browser.
- **[Compliance]** POI mint requires `p_acks={declaration_ack:true, atb_ack:true}` on every mint, plus ≥1 doc per side on bilateral matches (no waivers).
- **[CI]** `scripts/check-edge-function-paths.mjs` runs in `prebuild` and fails CI if any `fetchEdgeFunction` / `functions.invoke` / `/functions/v1/...` path doesn't match a real `supabase/functions/<name>/` directory.

## [3.5.0] - 2026-04-15

### Counterparty Ratings + Account Self-Deletion

- **[Feature]** **Counterparty rating system** — derived four-pillar ratings (Reliability, Responsiveness, Compliance, Settlement) with bands (not stars), time-decay, sample-size guards, append-only signal ledger, admin-resolved appeals, no free-text reviews.
- **[Feature]** **Account self-deletion** — Desk → Settings → My Profile "Danger zone". `delete-account` edge function soft-deletes profile (`status='pending_deletion'`, PII anonymised, roles revoked), keeps `auth.users` for 30-day grace, audits `account.self_deleted`. Guards: sole-org-admin and active POI (`PENDING_APPROVAL` / `ELIGIBLE` / `COMPLETION_REQUESTED`).
- **[WaD]** New `awaiting_attestations` status added to WaD lifecycle. `src/lib/wad-state.ts` lists it with the same affordances as `draft` (attest / seal / view_evidence). Build-time `check:wad-statuses` script validates `WAD_STATUSES` against the DB CHECK constraint.

## [3.4.0] - 2026-04-08

### Test-Mode Bypass + Webhook Replay Protection

- **[Admin]** **Test-mode bypass** — admin flags in `admin_settings.test_mode_bypass` short-circuit IDV / sanctions / KYB / UBO / ATB. Every use audited as `test_mode.bypass_used`. Global `TestModeBanner` mounted in `App.tsx`.
- **[Security]** **Webhook replay protection** — `webhook_replay_guard` ledger + `assertNotReplayed` helper. Stable 409 / `WEBHOOK_REPLAY` response. Wired into `auth-email-hook`, `handle-email-suppression`, `_shared/webhooks.ts`. Daily prune via `lifecycle-scheduler`.
- **[Observability]** **Session failure metrics** — client-side `localStorage` counters for `UNAUTHORIZED` / `REFRESH_FAILED` / `HEALTH_CHECK_FAILED` with last `requestId` + context. Wired in `edge-invoke` `EdgeInvokeError` + `AuthContext` health loop. Live `izenzo:session-failure-metric` window event.
- **[Email]** **Subject length contract** — all outbound email/Slack subjects pass through `clampSubject()` (200-char ceiling, preserves trace tail). Raw template-string concatenation of free-text into subject lines is forbidden.

## [1.3.0] - 2025-12-03

### ⚠️ Terminology Update: "Settle" → "Confirm Intent"

This release updates all terminology to clearly distinguish between **binding intent confirmation** and **non-binding exploration**.

#### Key Changes

| Before | After |
|--------|-------|
| "Settle Match" | "Confirm Intent" |
| "Settlement" | "Intent Confirmation" |
| "Settled" status | "Confirmed" (display only, DB still uses `settled`) |

#### What Creates Records?

| Action | Creates Audit/Evidence? |
|--------|------------------------|
| **Confirm Intent** | ✅ Yes - immutable proof of interest |
| Skip / Not Now | ❌ No - exploration only |
| Maybe Later | ❌ No - exploration only |
| View / Browse | ❌ No - exploration only |

#### Important Notes

- **No legal obligation**: "Confirm Intent" signals serious interest so the seller can prepare final terms. It does NOT create any contract, payment, or legal commitment.
- **Only Confirm creates records**: All other UI options (skip, maybe later, etc.) are soft behavioral signals that do NOT write to the database.
- **Backward compatible**: The API endpoint remains `/match/:id/settle` for compatibility, but now triggers `intent.confirmed` webhook event (also `match.settled` for legacy support).
