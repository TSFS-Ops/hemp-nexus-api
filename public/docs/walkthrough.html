Purpose: This document proves the system works as a system - not as isolated features. It walks through the complete lifecycle from zero to a sealed, evidence-backed trade record.

Lifecycle: Onboard → Verify → Discover → Engage → Mint POI → Complete → Seal WaD → Export Evidence

Terminology: This platform uses Counterparty, Trade Request, Proof of Intent (POI), and WaD (always written "Without a Doubt" — never "Warrant of Diligence"). We never use "Bid/Offer".

Billing: Platform credits are USD-native since 2026-05-01. 1 credit = $1.00 USD. Trade-side currencies on a Trade Request are commercial terms, not billing.

Phase 1 - Entity Onboarding & Due Diligence (~2 min)

1 Create Organisations (Buyer + Seller)

Two organisations are created to represent the trading parties.

NameAcme Pharma (Pty) LtdMedSupply Holdings

Each org gets a token_balances record initialised at 1,000 tokens.

POST /functions/v1/orgs

2 Register Entities + UBOs + Authority-to-Bind

For each organisation:

  1. Register a legal entity (company type, jurisdiction ZA)
  2. Register natural persons as UBO links with ownership ≥100%
  3. Create ATB records linking person → company (board_resolution)
Hard-gate: UBO ownership must sum to ≥100% per entity. ATB must be verified.
POST /functions/v1/entities
POST /functions/v1/authority-bind

3 Upload KYC Documents

Upload ID, registration certificate, UBO declaration to private kyc-documents bucket.

Storage path: {org_id}/kyc/{entity_id}/{filename}

4 Screen UBOs for Sanctions & PEP

Screen all natural persons linked as UBOs for both organisations.

Expected: All screenings return status: "clear"

Hard-gate: Screening must be within 30 days for WaD issuance.
POST /functions/v1/dilisense-screen

5 Compute Risk Scores

Risk scores based on: entity verification, UBO completeness, screening results, jurisdiction risk, document completeness.

Expected: risk_band: "low" or "medium"

Hard-gate: high or critical risk bands block WaD issuance.
POST /functions/v1/due-diligence

6 Approval Workflow

Multi-role approval based on trade value thresholds:

POST /functions/v1/due-diligence (action: approve)

7 Trade Approval Certification

Mark both organisations as "Approved to Trade". One-time certification, must remain valid at collapse time.

Checks: Entity ✓ · UBO 100% ✓ · ATB ✓ · Screening clear ✓ · Risk acceptable ✓ · Approval complete ✓

POST /functions/v1/trade-approval

Phase 2 - Discovery & Matching (~1.5 min)

8 Create a Trade Request

Each side opens a Trade Request — the canonical, persistent record of trade intent. Trade Requests live in the trade_requests table and survive across counterparty attempts: if an engagement is declined or expires, the parent Trade Request remains and can be re-engaged with a different counterparty without re-keying.

Buyer Trade Request: Paracetamol API 500mg (HS 2924.29), 50,000 kg, origin ZA → destination ZA, ZAR 15.00/kg DDP (commercial currency, not billing)

Seller Trade Request: Paracetamol Active Pharmaceutical Ingredient (HS 2924.29), 100,000 kg, origin ZA

Commodity values come from the static commodity taxonomy module (80+ HS-coded items). Trade Requests are non-binding — no tokens burned, no commitments made.

POST /functions/v1/trade-requests

9 Discovery (role-inverted search)

The discovery engine surfaces compatible counterparties using search role inversion: a buyer's Trade Request searches the seller pool, and vice versa. The 4-layer pipeline runs Internal book → Order book → AI semantic match → ILIKE fallback, with contact details masked until engagement.

Candidates are scored on commodity/HS-code similarity, jurisdiction compatibility, quantity overlap, and price-range fit. Search activity is captured in discovery_search_logs.

POST /functions/v1/search
POST /functions/v1/sr-discover

10 Engage Counterparty (hold-point)

The buyer org initiates an engagement with the seller org against the discovered Trade Request.

  • A match row is created as a child of the parent trade_requests row (linked via trade_request_id)
  • Engagement enters pending status — the seller must accept before any POI can be minted
  • Until acceptance, all POI mint attempts return 409 / ENGAGEMENT_PENDING
  • Dual-path notification fires (admin + counterparty user) via Resend, with subjects clamped to 200 chars by clampSubject()
POST /functions/v1/poi-engagements

11 Counterparty Accepts Engagement

The seller org accepts the engagement.

  • atomic_accept_bind runs (service_role-only, since SECDEF Stage D1)
  • Engagement status → accepted
  • Hold-point clears; the buyer can now progress to POI mint
  • Audit log records engagement.accepted
POST /functions/v1/poi-engagements (action: accept)

Phase 3 - POI Lifecycle & Mint (~2 min)

12 Pre-flight Validation + Probability

Pre-flight checks: Both parties approved ✓ · Approval not expired ✓ · No open compliance cases ✓ · Org not frozen ✓ · No global freeze ✓

POI Probability calculated from 7 weighted factors:

FactorWeightExpected
Entity verification20%1.0
UBO completeness15%1.0
ATB verification10%1.0
Screening clear15%1.0
Trade approval15%1.0

Hard-gate: Probability < 50.1% blocks POI mint.
POST /functions/v1/preflight
POST /functions/v1/poi-probability

13 POI Mint (Binding Event)

POI mint runs through atomic_generate_poi_v2 — a service_role-only PostgreSQL function (SECDEF Stage D1, 2026-04-22). The browser never calls it directly; it goes through the owning edge function.

Mandatory request fields: org_id, counterparty_org_id, trade_request_id, commercial terms (quantity, price, currency, incoterms), idempotency_key (header), and p_acks={declaration_ack:true, atb_ack:true} on every mint.

Validations enforced:

  1. Engagement is accepted (else 409 / ENGAGEMENT_PENDING)
  2. No active dispute on the match (else 409 / DISPUTE_ACTIVE)
  3. Both parties "Approved to Trade" with valid approval
  4. Dynamic approval tier met (5-factor risk scoring)
  5. Intent state is ELIGIBLE or COMPLETION_REQUESTED
  6. POI completion probability ≥ 50.1%
  7. Acknowledgements present (declaration_ack + atb_ack)
  8. For bilateral matches: ≥1 document per side (no waivers)
  9. PostgreSQL advisory lock acquired (concurrency control)
  10. Idempotency (duplicate → returns original record)

Evidence Strength Indicator: documents are not individually mandatory beyond the per-side minimum. The UI surfaces a red→amber→green bar — more documents = stronger bundle. The wizard never auto-skips the Match step; the strict audited waiver dialog blocks POI mint when zero docs and zero notes.

Recorded: POI row with state → COMPLETED · atomic_token_burn deducts USD-native cost (exempt_burn for founder/admin) · SHA-256 payload hash · Append-only audit_logs entry

POST /functions/v1/poi-mint

Phase 4 - Evidence & Certification (~1.5 min)

14 Generate Evidence Pack v1

Tamper-evident bundle with canonical JSON (sorted keys, no whitespace) for deterministic SHA-256 hashing.

Includes: full payload, signature status, timestamps, approval chain, audit trail.

POST /functions/v1/evidence-pack

15 Create WaD (Without a Doubt) Certificate

The WaD layer enforces 10 deterministic hard-gates:

#GateRequirement
1Intent stateCOMPLETED
2Entity statusVERIFIED
3UBO integrity100% verified
4ATB recordsActive & verified
5Governance docsAll mandatory validated
6Compliance casesZero open
7Token balanceSufficient for burns
8ScreeningWithin 30 days
9Risk bandNot high/critical
10Webhook connectivityNeither party's primary webhook endpoint is auto-disabled
POST /functions/v1/wad

16 Attestations (Buyer + Seller)

WaD enters awaiting_attestations. Both parties sign: "I confirm this is not a contract. No payment. No obligation. This is a record that intent was confirmed."

POST /functions/v1/wad/{wadId}/attest

17 Seal the WaD

With both attestations, the WaD is sealed: canonical payload hashed (SHA-256), ledger entry hash chains to previous WaD, status → sealed.

POST /functions/v1/wad/{wadId}/seal

18 Download Certificate

Export sealed WaD certificate: transaction summary, party details, attestations, seal hash, ledger entry hash, evidence bundle.

GET /functions/v1/wad/{wadId}/certificate

Verification Checklist

  • Two organisations exist with status: active
  • Entities registered with UBO ownership ≥ 100%
  • ATB records verified for both parties
  • Screening results clear and < 30 days old
  • Risk bands are low or medium
  • Both orgs "Approved to Trade"
  • Trade Requests created (buyer + seller)
  • Engagement requested and accepted (hold-point cleared)
  • POI minted via atomic_generate_poi_v2 (USD-native cost burned)
  • POI probability ≥ 50.1%
  • POI record in append-only audit_logs
  • SHA-256 payload hash recorded
  • NTP drift ≤ 1000ms (hardened status)
  • WaD passes all 10 hard-gates (incl. webhook connectivity)
  • Both parties attested via awaiting_attestations path
  • WaD sealed with SHA-256 hash chain
  • Evidence pack exportable
  • Audit trail complete (15+ events)
  • Token balance reflects USD-native burns
